## 1. Executive Declaration

**아날로그의 역습, 데이터의 지배.**

GOHSY FASHION은
80년대 일본의 아날로그 패션 황금기 자산을
AI 기반 시스템 공학으로 정형화하여
남성 스타일을 **"인출 가능한 데이터"**로 만드는 프로젝트다.

이것은 쇼핑몰이 아니다.

> 패션 디자인 4년 커리큘럼을
> 1,000개의 스타일 모듈로 역공학하고
> 이를 상업 엔진과 결합하는 패션 테크 시스템이다.

경쟁자는 Pinterest를 본다.
우리는 구조를 채굴한다.

## 2. 전략적 포지션

### 우리는 무엇을 하는가

1. 스타일을 감각에서 분리한다.
2. 스타일을 수치화한다.
3. 스타일을 공식화한다.
4. 스타일을 데이터로 축적한다.
5. 스타일을 매출로 연결한다.

### 왜 남성인가

| 이유 | 근거 |
|------|------|
| 공급 부족 | 남성 패션 콘텐츠 시장은 여성 대비 1/5 수준 |
| 높은 충성도 | 남성 구매 패턴: 탐색 짧고 브랜드 충성 높음. 한번 잡으면 반복 구매 |
| 데이터 소스 일치 | 80-90 아카이브 = 남성 잡지 기반. 소스와 타겟이 같다 |
| 코디 수요 | "뭘 입어야 할지 모르겠다" = 남성 패션 최대 통증점 |

### 전략적 정의

스타일 = 계급 언어.
스타일링 = 과거 유산의 현대적 재조합.
상품 = 콘텐츠의 파생 결과물.

> **상품은 옷이 아니라 "사회적 문장"이다.**

GOHSY는 무질서한 잡화몰이 아니라:

> **"남성 스타일 리팩토링 스튜디오"**

## 3. 타겟 고객

### 3.1 1차 타겟: "옷 잘 입고 싶은데 방법을 모르는 남자"

| 항목 | 정의 |
|------|------|
| 연령 | 25-40세 |
| 소득 | 월 250-500만원 (직장인, 자영업) |
| 특성 | 패션에 관심은 있으나 무신사/29cm 탐색이 귀찮음 |
| 구매 행동 | 검색 → 영상 확인 → 즉시 구매 (의사결정 빠름) |
| 통증점 | "뭘 사야 할지 모르겠다", "코디를 못하겠다" |

### 3.2 왜 GOHSY에서 사는가

쿠팡에 같은 옷이 수천 개 있다. 차이점:

| 일반 셀러 | GOHSY |
|-----------|-------|
| 상품 사진 1장 | 룩북 + 웹툰 + 코디 영상 + 스타일 공식 해설 |
| "이 옷 어때요" | "이 옷을 이렇게 입어라" (스타일 공식 기반) |
| 상품 나열 | 코디 솔루션 제안 |
| 감각 의존 추천 | V-F-C-D 데이터 기반 추천 |

**판매하는 것은 옷이 아니라 "코디 솔루션"이다.**

### 3.3 2차 타겟: "복고 감성에 반응하는 남자"

- 80-90년대 일본 시티팝/아메카지 감성
- 유튜브에서 "빈티지 패션", "아메카지" 검색하는 층
- 이 층은 콘텐츠 충성도가 높고, 팬덤화 가능
- **Style Master 콘텐츠의 핵심 소비자**

## 4. 시장 분석

### 4.1 범위

남성 패션 전 카테고리:

1. 스트릿
2. 클래식/수트
3. 캐주얼
4. 워크웨어
5. 아카이브 복고

**초기 집중: 캐주얼 + 아카이브 복고** (아카이브 DB와 직결되는 카테고리)

### 4.2 경쟁자 분석

| 경쟁자 | 강점 | 약점 | GOHSY 대비 |
|--------|------|------|------------|
| 무신사 | 브랜드 집합, 트래픽 | 자체 콘텐츠 약함, 남성 비중 감소 | 무신사에 없는 코디 콘텐츠 + 스타일 공식 |
| 29cm | 큐레이션, 감성 | 여성 중심, 객단가 높음 | 남성 특화 포지션 |
| 쿠팡 패션 셀러 | 가격 경쟁력, 트래픽 | 콘텐츠 없음, 상품 사진만 | 콘텐츠 양 10배 차이 |
| 에이블리/지그재그 | MZ 여성 특화 | 남성 없음 | 시장 겹침 없음 |
| YouTube 패션 채널 | 구독자, 영향력 | 직접 판매 안 함 | 콘텐츠+판매 통합 |
| 스타일쉐어 | 커뮤니티 | 수익화 미약 | 데이터 → 상품 루프 |

**포지션**: 쿠팡의 가격 + 무신사의 큐레이션 + YouTube의 콘텐츠를
**1인 AI 시스템 + Style Master Engine**으로 구현.

### 4.3 차별성 매트릭스

| 항목 | 일반 셀러 | GOHSY |
|------|-----------|-------|
| 스타일 근거 | 감각, 트렌드 따라가기 | 80s 아카이브 구조 분석 (V-F-C-D) |
| 상품 촬영 | 스튜디오 촬영 후 업로드 | AI 합성 대량 생성 |
| 콘텐츠 비율 | 1상품 1이미지 | 1상품 10콘텐츠 (룩북+웹툰+영상+도면) |
| 코디 추천 | 에디터 감각 의존 | 스타일 공식(Formula) 기반 자동 생성 |
| 운영 방식 | 재고 중심 | 콘텐츠 중심 |
| 의사결정 | 감각 의존 | 판매 데이터 → 공식 진화 |
| 인력 | 3-5인 | 1인 + AI |

## 5. Style Master Engine — 핵심 기술 자산

이것이 GOHSY의 본체다.
옷을 파는 게 아니라 **스타일 구조를 엔진화**하는 것이다.

### 5.1 Analog Moat — 왜 80년대 일본인가

| 이유 | 설명 |
|------|------|
| 버블 경제 정점 | 소비력 극대화 시기. 패션 투자 = 최고 수준 |
| 에디터/포토 디렉터 완성도 | 잡지 문화 정점. 촬영+레이아웃+스타일링 = 종합 예술 |
| 동양인 체형 기반 | 서양 패션지 데이터와 달리 한국인 체형에 직접 적용 가능 |
| 디지털화 미완료 | Dark Data 영역. 경쟁자가 접근하려면 같은 아카이브를 구해야 함 |

이것은 단순 아카이브가 아니다.

> **"검증된 미적 정답지 + 디지털 공백 영역" = Analog Moat**

### 5.2 V-F-C-D Meta Schema

스타일을 감각이 아니라 데이터로 다루기 위한 4계층 구조:

#### V — Variables (정량 변수)

1개 룩(Look)을 수치로 분해한다:

| 변수 | 타입 | 범위 | 설명 |
|------|------|------|------|
| `upper_length_ratio` | float | 0.0-1.0 | 상의 길이 비율 (전체 대비) |
| `pant_length_ratio` | float | 0.0-1.0 | 바지 길이 비율 |
| `pant_width_category` | enum | Slim/Straight/Wide | 바지 폭 카테고리 |
| `layer_depth` | int | 1-4 | 레이어링 깊이 (이너/미드/아우터/오버) |
| `contrast_index` | float | 0.0-1.0 | 상하의 명도/채도 대비 (HSV 기반) |
| `shoe_exposure_rate` | float | 0.0-1.0 | 발목 노출 비율 |
| `silhouette_type` | enum | A/I/Y/Box | 실루엣 유형 |
| `color_palette` | array | hex[] | 사용 색상 목록 |
| `season` | enum | SS/FW/Transition | 시즌 |
| `formality_score` | float | 0.0-1.0 | 격식도 (캐주얼 0 ~ 포멀 1) |

#### F — Formulas (조합 알고리즘)

변수 조합에서 패턴을 추출한다. **아카이브에서 반복 검증된 공식만 등록:**

```
Formula #001: "Y-Line Balance"
  IF silhouette_type == "Y"
  THEN upper_length_ratio ≤ 0.35
   AND pant_width_category IN [Slim, Straight]
   AND contrast_index ≥ 0.3

Formula #002: "Wide Pant Anchor"
  IF pant_width_category == "Wide"
  THEN shoe_exposure_rate ≤ 0.25
   AND layer_depth ≤ 2
   AND upper_length_ratio ≤ 0.30

Formula #003: "Layer Stack"
  IF layer_depth ≥ 3
  THEN contrast_index ≤ 0.40
   AND color_palette.length ≤ 4
   AND formality_score ≤ 0.6

Formula #004: "Monochrome Authority"
  IF contrast_index ≤ 0.15
  THEN silhouette_type IN [I, A]
   AND formality_score ≥ 0.5

Formula #005: "City Boy Summer"
  IF season == "SS"
   AND layer_depth == 1
  THEN upper_length_ratio ≤ 0.30
   AND shoe_exposure_rate ≥ 0.6
   AND pant_length_ratio ≤ 0.85
```

**이 공식은 가정이 아니다. 아카이브에서 추출한 뒤 판매 데이터로 검증한다.**
Phase 1에서 10개 추출 → Phase 3에서 50개 → Phase 5에서 100개+.
판매 전환율이 낮은 공식은 수정하거나 폐기한다.

#### C — Context (계급/문화 코드)

스타일은 옷이 아니라 **사회적 신호**다:

| 코드 | 원류 | 핵심 변수 조합 | 타겟 연결 |
|------|------|----------------|-----------|
| Ivy | 미국 아이비리그 → 일본 재해석 | formality 0.6+, layer_depth 2, contrast 중간 | 전문직 30대 |
| City Boy | 도시 전문직 청년 | formality 0.3-0.5, season SS, layer 1-2 | 직장인 25-30 |
| Workwear | 노동/남성성 코드 | pant_width Wide, layer 2+, formality 0.2-0.4 | 자영업/크리에이터 |
| Amekaji | 미국 캐주얼 일본화 | 모든 변수 중간값, contrast 낮음 | 복고 감성 전 연령 |

Context는 **상품 기획 + 콘텐츠 톤 + YouTube 시리즈 매핑**에 사용한다.
같은 상품도 Ivy Context면 "주간 비즈니스 코디", City Boy Context면 "주말 브런치 코디"로 콘텐츠가 달라진다.

#### D — Data Loop (피드백 엔진)

공식은 **판매 데이터로 진화**한다:

| 데이터 | 소스 | 피드백 대상 |
|--------|------|-------------|
| CTR (클릭률) | 쿠팡/네이버 | 썸네일 + 상품명 공식 |
| Conversion Rate | 쿠팡/네이버 | 상세페이지 구성 + 가격대 |
| Return Rate | CS 데이터 | 사이즈/핏 공식 (Variables 보정) |
| 체류 시간 | YouTube/gohsy.com | 콘텐츠 포맷 + 영상 길이 |
| 재구매율 | 쿠팡/네이버 | Context 매칭 정확도 |
| Price Elasticity | A/B 가격 테스트 | 가격대별 전환 최적점 |

```
분기별 루프:
  판매 데이터 수집 → 공식별 성과 분석 → 저성과 공식 수정/폐기
  → 고성과 공식 파생 → 다음 분기 상품 기획에 반영
```

### 5.3 V-F-C-D → PRODUCT_CARD 연결

Style Master Engine은 독립 시스템이 아니다.
기존 4채널 파이프라인에 직결된다:

```
V-F-C-D Schema
     │
     ▼
PRODUCT_CARD.json (기존 SoT)
  └── style_module (신규 필드)
       ├── variables: { silhouette_type: "Y", contrast_index: 0.35, ... }
       ├── formula_ids: ["F001", "F005"]
       ├── context: "city_boy"
       └── source_archive: "popeye_1987_vol23_p42"
     │
     ▼
기존 파이프라인 그대로 작동:
  AI 에셋 생성 → GPU 렌더 → 4채널 배포 → 매출 → D(Data Loop) → 공식 진화
```

**기존 파이프라인을 바꾸는 게 아니라, PRODUCT_CARD에 `style_module` 필드를 추가하는 것이다.**

| **SKU당 비용** | **30-50만원** | **1,000-5,000원** |

기존 대비 **99% 비용 절감**. "0원"이 아니라 "99% 절감"이다.
API 호출, 큐레이션 시간, 품질 검수 시간은 0이 아니다.

### 6.3 고씨 비주얼 앵커

AI 합성만으로는 "가짜 느낌"을 지울 수 없다.

- AI 이미지만 → 신뢰 부족. 실물 사진이 섞여야 전환율 올라감
- 고씨 = 브랜드 얼굴. 일관된 비주얼 아이덴티티
- AI 모델 + 실물 앵커 혼합 = "이 브랜드는 실체가 있다"

이건 복제 불가. 사람은 복사가 안 된다.

       │

       │

- 네이버 쇼핑 상위 노출 = 자연 유입

### 7.3 YouTube — 트래픽 엔진

| 시리즈 | 포맷 | 빈도 | Style 연결 |

## 8. 수익 모델

### 8.1 1단계 — 위탁/POD (재고 0)

| 항목 | 구조 |

| 소싱 | 위탁 판매 (동대문/도매) 또는 POD |

검증 단계에서 재고를 쌓는 건 자살 행위다.
데이터가 쌓인 후에 자체 기획으로 전환한다.

### 8.2 2단계 — 데이터 기반 자체 기획

| 조건 | 전환 기준 |
|------|-----------|
| 전환 시점 | 월 100건 이상 판매 SKU 확인 후 |
| 자체 기획 | 판매 데이터 + Style Formula 조합 |
| 마진 구조 | 원가 20-30% → 소비자가 대비 마진 50-60% |

### 8.3 3단계 — Style Master 콘텐츠 수익화

| 채널 | 수익원 |
|------|--------|
| YouTube 시리즈 E | 광고 수익 + 멤버십 |
| gohsy.com Style DB | 유료 열람 or 프리미엄 코디 추천 |
| 코디 컨설팅 | Formula 기반 1:1 스타일링 |

**이 단계는 Phase 4 이후. 검증 전에 수익 모델로 잡지 않는다.**
Phase 3까지는 상품 판매 수익만 본다.

### 8.4 4단계 — Style Query Engine (B2B 가능성)

```
입력: "키 178, 보통 체형, 30대 전문직, 봄, 캐주얼"
출력: Formula 조합 + 추천 코디 + 상품 링크
```

이것은 API로 제공 가능하다. 다른 쇼핑몰이 가져다 쓸 수 있다.
**Phase 5 이후 검토. 현재 수익 모델에 포함하지 않는다.**

### 8.5 수치 시뮬레이션

**전제 조건 (보수적):**

| 항목 | 값 | 근거 |
|------|-----|------|
| SKU | 30개 (초기) | 위탁 기준 관리 가능 수량 |
| 객단가 | 39,000원 | 캐주얼 남성복 평균 |
| 일 트래픽 | 500 (쿠팡+네이버 합산) | 신규 셀러 초기 3개월 평균 |
| 전환율 | 0.8% | 신규 셀러 초기 (업계 평균 2% 대비 보수적) |
| 마진율 | 20% | 위탁 기준 |

```
초기 (Month 1-3):
일 주문: 500 × 0.008 = 4건
월 매출: 4 × 39,000 × 30 = 4,680,000원
월 순이익: 4,680,000 × 0.20 = 936,000원
```

**6개월 후 (트래픽 + SKU + 리뷰 축적):**

| 항목 | 값 |
|------|-----|
| SKU | 100개 |
| 일 트래픽 | 2,000 (YouTube 유입 포함) |
| 전환율 | 1.2% (리뷰 축적 효과) |
| 객단가 | 45,000원 (자체 기획 혼합) |
| 마진율 | 30% (자체 기획 비중 증가) |

```
6개월 후:
일 주문: 2,000 × 0.012 = 24건
월 매출: 24 × 45,000 × 30 = 32,400,000원
월 순이익: 32,400,000 × 0.30 = 9,720,000원
```

**이 수치는 매월 실제 데이터로 교체한다. 가정은 가정일 뿐이다.**

## 9. 방어벽 (Defensibility)

AI 자체는 방어벽이 아니다. 누구나 쓸 수 있다.

### 9.1 진짜 방어벽 4개

**① Analog Moat — 80s 아카이브 구조 DB**

디지털화되지 않은 고급 아카이브 데이터.
경쟁자가 같은 잡지를 구해서, 같은 분석을 하고, 같은 시간을 써야 한다.
구매 데이터는 쿠팡이 더 많다.
**스타일 구조 데이터**는 GOHSY만 쌓는다.

**② Quantified Style DB — 수치화된 스타일 공식**

감각이 아닌 V-F-C-D 스키마.
공식 1개를 만드는 데 아카이브 분석 + 판매 검증이 필요하다.
100개 공식 = 100번의 루프. 시간이 곧 해자.

**③ Real Visual Anchor — 고씨 실물**

AI 합성만으로는 신뢰 불가. 실물 앵커가 브랜드를 실체화한다.
사람은 복사가 안 된다.

**④ Data Feedback Evolution — 판매 데이터 기반 진화**

공식이 판매 데이터로 진화한다. 틀린 공식은 수정되고, 맞는 공식은 파생된다.
이 루프는 시간이 갈수록 정확해진다. 후발 주자는 Day 1부터 시작해야 한다.

### 9.2 방어벽이 아닌 것 (착각하면 죽는다)

| 착각하기 쉬운 것 | 왜 방어벽이 아닌가 |
|-------------------|---------------------|
| "AI를 쓴다" | 경쟁자도 다 쓴다 |
| "자동화했다" | 도구는 누구나 산다 |
| "GPU가 있다" | 클라우드로 빌리면 된다 |
| "쿠팡/네이버 매출 데이터" | 그들이 수억 건 갖고 있다 |
| "스타일을 안다" | 감각은 복제된다. 공식만 복제 안 된다 |

## 11. 병목 진단

**AI도 Style Master Engine도 병목이 아니다.**

진짜 병목:

```
① 소싱    — 좋은 위탁처 확보 = 상품 품질 = 반품률 = 리뷰 = 매출
② 물류    — 위탁이면 위탁처 품질, 자체면 3PL 계약
③ 반품    — 온라인 의류 반품률 20-30%. 이걸 견딜 CS 체력
④ CS      — 1인 운영에서 CS 부하 = 시스템 멈춤
⑤ 정책    — 쿠팡/네이버 정책 변경에 즉각 대응하는 민첩성
```

이걸 해결하지 못하면 GPU도 Style DB도 장난감이다.

| 병목 | 원칙 |
|------|------|
| 소싱 | 3곳 이상 위탁처 확보 → 단일 의존 방지 |
| 물류 | 위탁으로 시작 → 데이터 쌓이면 3PL |
| 반품 | SKU 30개 제한 → CS 부하 관리 가능 범위 유지 |
| CS | 자동 응답 템플릿 + FAQ 페이지 → 1인 처리 가능 구조 |
| 정책 | adapter 패턴 → 플랫폼 변경 시 adapter만 수정 |

## 15. 최종 정의

GOHSY FASHION은:

> 아날로그 황금기의 스타일을 데이터 알고리즘으로 정형화하고,
> AI와 결합하여 남성의 스타일을 설계하는 패션 엔진 시스템.

유행을 쫓지 않는다. 구조를 설계한다.

이것이 완성되면:

```
Archive → Style DB (V-F-C-D) → AI Synthesis → 4-Channel Market
→ Sales Data → Refined Formula → Next Drop → ...
```

이 루프가 스스로 돌아간다.
